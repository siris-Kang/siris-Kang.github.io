Mausoleum 영묘

탈출을 목표로 협력과 배신을 반복하는 3D 멀티플레이 추리 게임

0. 프로젝트 소개

mausoleum 소개 자료

mausoleum GitLab

1. 프로젝트 개요

  • 보물을 훔치기 위해 고대의 성에 잠입한 도굴꾼들은 성에 갇히고 맙니다.
  • 플레이어는 성 내부에 잠입한 도굴꾼이 되어 보물과 아이템을 수집하고, 단계별 관문을 통과해 탈출해야 합니다.
  • 악의 수치에 따라 플레이어 관계가 계속 바뀌도록 설계해, 단순 협동이 아니라 심리전과 전략적 선택이 반복되도록 기획했습니다.

주요 기능

  • 무기 / 방패 / 보물 / 도구 아이템 시스템
  • 악의 획득 및 공개 인카운터
  • 고해성사, 저주받은 검 등 개인 이벤트
  • 제물, 난투, 심판 등 단체 이벤트
  • 정원 → 안개 구역 → 폐쇄된 성당 → 외나무다리로 이어지는 스테이지 구성

2. 담당 역할

  • C++ 기반 아이템 시스템 구조 설계
  • 아이템 공통 인터페이스 및 사용 흐름 구현
  • 퀵슬롯과 아이템 사용 로직 분리
  • UI 표시 정보와 실제 게임 상태가 맞지 않는 구간 점검 및 수정
  • 공용 Blueprint/맵 에셋 협업 규칙 정리

팀 내 역할 분담

  • 팀은 클라이언트, 서버, 레벨/에셋 작업을 나누어 진행
  • 나는 아이템 시스템 중심의 클라이언트 구현을 담당

3. 사용 기술

Unreal Engine 5

  • 3D 멀티플레이 게임, 레벨 구성, UI, 상호작용 시스템을 통합적으로 개발하기 위해 사용했습니다.

C++

  • 아이템 종류가 계속 늘어나는 구조였기 때문에, 공통 인터페이스와 책임 분리를 명확히 하기 위해 사용했습니다.

Blueprint

  • UI 조정, 에셋 설정, 레벨 배치처럼 빠른 확인과 반복 수정이 필요한 작업에 활용했습니다.

GitLab

  • 브랜치 / MR 기반 협업과 작업 이력 관리를 위해 사용했습니다.

Jira

  • 기획 변경이 잦은 프로젝트여서 작업을 짧은 단위로 나누고 우선순위를 관리하기 위해 사용했습니다.

4. 구현 사항

  • 아이템이 어떤 클래스인지 몰라도 사용할 수 있는 공통 인터페이스 구조를 설계했습니다.
  • 무기 / 방패 / 보물 / 도구를 하나의 흐름 안에서 처리할 수 있도록 구현했습니다.
  • 퀵슬롯이 아이템의 구체 타입에 직접 의존하지 않도록 구조를 분리했습니다.
  • 아이템 추가 시 기존 코드 수정 범위를 줄일 수 있도록 모듈화했습니다.
  • UI에 표시되는 정보와 실제 게임 상태 반영 흐름을 점검하고 수정했습니다.
  • 협업 중 공용 Blueprint와 맵 에셋 충돌을 줄이기 위한 작업 방식을 정리했습니다.

5. 문제 해결 사례

1) 아이템 인터페이스 구조 설계

상황

무기, 방패, 보물, 도구처럼 동작이 다른 아이템을 하나의 시스템 안에서 처리해야 했습니다. 초기에 단순 분기문 위주로 구현하면 아이템이 늘어날수록 구조가 복잡해지고, 수정 범위도 계속 커질 문제가 있었습니다.

해결

아이템별 구현보다 먼저 공통 인터페이스와 책임 분리 구조를 설계했습니다. 퀵슬롯은 선택된 아이템을 사용한다는 요청만 보내고, 실제 동작은 각 아이템이 담당하도록 흐름을 나눴습니다. 이를 통해 슬롯 시스템이 개별 아이템 클래스에 직접 의존하지 않도록 만들었습니다.

성과

아이템이 추가되거나 기능이 수정될 때 퀵슬롯 전체를 다시 고치지 않아도 되었습니다. 기능 확장 시 수정 범위를 줄이고, 아이템 시스템을 더 안정적으로 관리할 수 있었습니다.

의미

변경 가능성이 큰 게임 시스템은 기능 구현보다 먼저 인터페이스와 책임 구조를 잡는 것이 중요하다는 점을 배웠습니다. 이 경험을 통해 유지보수성과 확장성을 고려한 구조 설계 기준을 정리할 수 있었습니다.

2) Unreal Blueprint 에셋 GitLab 협업 충돌

상황

Unreal의 맵, 캐릭터 Blueprint 같은 에셋은 텍스트가 아니라 바이너리 파일이라, 작은 수정만 있어도 GitLab에서 충돌이 발생했습니다. 특히 공용 맵이나 공용 Blueprint를 여러 명이 동시에 수정하면 병합이 어렵고, 작업 내용이 꼬일 위험이 컸습니다.

해결

맵 작업은 각자 개인 맵을 따로 만들어 기능을 확인한 뒤 반영하도록 했습니다. 공용 Blueprint는 한 번에 한 명만 수정하는 원칙을 세웠고, 급하게 반영해야 하는 작업은 다른 팀원이 수정 중인지 먼저 확인한 뒤 바로 MR을 올리는 방식으로 운영했습니다.

성과

공용 에셋 충돌로 인한 재작업이 줄었고, 맵과 캐릭터 관련 작업을 비교적 안정적으로 병렬 진행할 수 있었습니다. 협업 중단이나 병합 실패로 인한 일정 손실도 줄일 수 있었습니다.

의미

협업은 Git 사용 자체보다, 엔진과 파일 형식의 특성에 맞는 작업 규칙을 함께 설계하는 것이 중요하다는 점을 배웠습니다. 특히 Unreal처럼 바이너리 에셋 비중이 큰 환경에서는 기술적 한계를 팀 규칙으로 보완해야 한다는 기준을 얻었습니다.

6. 성과 및 결과

  • 협동, 배신, 탈출 요소가 결합된 3D 멀티플레이 추리 게임 안에서 핵심 아이템 시스템을 구조적으로 구현했습니다.
  • 아이템 공통 인터페이스와 퀵슬롯 분리를 통해 새 아이템 추가 시 수정 범위를 줄일 수 있는 구조를 확보했습니다.
  • Unreal 바이너리 에셋 협업 규칙을 정리하며 공용 Blueprint와 맵 작업을 더 안정적으로 병렬 진행할 수 있었습니다.
